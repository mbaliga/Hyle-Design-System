# Roadmap, limitations & decisions

## Known limitations and tradeoffs

**Facet is not true low-poly.** It quantises a noise field into stepped contours, giving lumpy plates with crack-seams — it reads as cracked stone, not the razor-straight polygonal facets of Gris's vector art. True faceting needs per-cell flat normals, which means a 3D Voronoi (on the order of 27 cell tests per sample) and a meaningfully heavier per-pixel cost. Open if the exact vector-facet edge quality matters more than the performance.

**Ruins-Once is SDF-heavy** and can drag on a phone. The colonnade, boulder, debris and poles stack up. Reduce sprite Count and Inner before anything else if it stutters.

**Displacement top frequency is capped at 60** (it was 74). The cap stops close-up pixelation but costs a little fineness at point-blank range. A distance-adaptive frequency would be the right fix; the naive attempt at distance attenuation flattened the texture and was reverted, so this needs doing properly if revisited.

**Dynamic uniform-array indexing** (`uSpr[si]` indexed by a loop variable) is implementation-defined in WebGL1. It works on the Adreno 740 development target. If a future target GPU rejects it, the on-screen error readout will name the offending line; the fix would be an unrolled loop or packing the sprite data into a texture instead of a uniform array.

**The sun disc elevation is fixed.** Aim rotates it around the horizon but cannot raise or lower it.

**Sprite world bounds are fixed** at roughly ±2.8 around the origin. In tall scenes such as Towers the wisps cluster near the base rather than ranging up the full height of the forms.

**No persistence.** All state is in memory; a refresh resets to defaults. Reseed only randomises the seed and re-initialises the sprites — it does not restore or save a configuration.

## Pending / requested, not yet built

A dedicated twist/swirl scene (the ferrofluid-spiral reference) and a parallel-rows-receding scene were both registered and not built. True polygonal faceting is the big open visual item (above). Smaller items: per-scene sprite bounds and a sprite count that scales with the scene; a sun-elevation control and an optional second light; and a two-colour smog gradient for richer aerial perspective.

## Ideas worth considering

A **preset system** is probably the highest-value next addition — the look-space is large and the good combinations (the BR2049 and Gris recipes in the README) are not discoverable by fiddling. Named presets that set the colour pickers, surface mode, Glow and Atmos together would make the engine usable without a tour.

Beyond that: exporting the current frame (`canvas.toDataURL`) and encoding the current state into the URL hash for shareable looks; and a reduced-step "mobile" quality toggle (lower `MAXS`, drop AO or shadow) for low-end GPUs.

## Decision log

Why the engine is shaped the way it is, so these are not re-litigated by accident:

A true SDF **raymarcher** was chosen over the earlier 2.5D-with-fake-normals approach because the monumental read needs real 3D depth, real normals, ambient occlusion and soft shadows.

The **three-colour model** (stone / smog / light) exists because single-tone tinting collapsed the form-versus-fog contrast into mush. Separating material, atmosphere and light is the brutalist key.

**Haze brightness is decoupled from Glow** so that dimming the light produces dark silhouettes against bright haze rather than dimming the whole frame — without this, the Blade Runner and Gris silhouette looks are not achievable.

There is **no ground-plane geometry** because a raymarched ground plane was both slow (grazing-angle rays take hundreds of micro-steps) and buggy (the displaced plane's hit point jittered and the surface noise crawled in tiles). Height-fog plus a bright low haze band replace it.

The sprites are **wisps, not spheres**, per an explicit art requirement, rendered as a two-sample directional glow.

The project is **generated by a Python script and validated in Node** because the file is large enough that shell-heredoc writes truncated it, and because the build environment has no GPU — the parse and balance checks are the only pre-device safety net.
